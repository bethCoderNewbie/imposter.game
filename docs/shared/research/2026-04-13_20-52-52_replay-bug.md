# Research: Replay Bug at Game Over

## Executive Summary

Three distinct bugs were found in the rematch ("Play Again") flow. The primary bug makes the mobile
"Play Again" button actively break auto-redirect when the host triggers a rematch. Two secondary bugs
leave stale display state from the ended game bleeding into the new game.

---

## Bug 1 — PRIMARY: Mobile "Play Again" Clears Session, Races with WS Redirect

### Location
`frontend-mobile/src/components/GameOverScreen/GameOverScreen.tsx:68`
`frontend-mobile/src/App.tsx:172`

### What Happens
The mobile `GameOverScreen` renders a "Play Again" button wired to:
```typescript
onPlayAgain={() => { clearSession(); setSession(null) }}
```

`clearSession()` removes `ww_session` from localStorage; `setSession(null)` drops the React session
state. The app re-renders → `session === null` → `<OnboardingForm>` is shown. The WebSocket
connection (which depends on `session.game_id`) is torn down.

### Why It Breaks the Rematch Flow

The correct rematch flow:
1. Host clicks "Play Again" on display → `POST /api/games/{id}/rematch`
2. Backend creates new game, broadcasts `{"type":"redirect","new_game_id":"...","players":{...}}`
3. Mobile's WS receives redirect → `handleRedirect` in `App.tsx:62` sets new session
4. `useGameState` detects `gameId` change → fence reset → WS reconnects to new game

**If the mobile player taps "Play Again" before the redirect arrives:**
- Session cleared → WS closed
- Redirect is broadcast to a dead socket → not received
- `rematch_redirect` is persisted in Redis (`endpoint.py:114`) for disconnected-player recovery,
  but the player has no session to reconnect with → replay never fires
- Player is stranded at OnboardingForm, must manually enter the new game code

**If redirect arrives before player taps the button:** works correctly.

### Why the `handleRedirect` guard doesn't save them
`App.tsx:68`: `setSession(prev => { if (!prev) return null … })` — if session was already cleared,
`prev` is null and the function returns null. The player is not migrated.

### Root Cause
The "Play Again" button should say "Leave Game". Tapping it is an intentional departure. The actual
"play again with same group" action is controlled exclusively by the display host. Mobile players
should **wait** for the host's redirect rather than having a button that races with it.

## Bug 2 — SECONDARY: Display `frozenVotes` Persists into New Game

### Location
`frontend-display/src/App.tsx:40` — `frozenVotes` state
`frontend-display/src/components/DayScreen/DayScreen.tsx:18` — usage

### What Happens
`frozenVotes` is set in `display/App.tsx` when `day_vote → !day_vote`:
```typescript
if (prev === 'day_vote' && curr !== 'day_vote') {
  setFrozenVotes({ ...gameState.day_votes })
}
```
It is never reset when `gameId` changes (rematch). The display's `useGameState` useEffect clears
`gameState` on `gameId` change, but `frozenVotes` is outside that hook.

In the new game, once `DayScreen` is rendered (phase `day` or `hunter_pending`):
```typescript
const votes = isVoting ? gameState.day_votes : (frozenVotes ?? {})
```
Since `isVoting` is false during `day` phase, it falls back to `frozenVotes` — which still holds the
**last round's vote snapshot from the previous game**. The `VoteWeb` SVG overlay (`DayScreen.tsx:46`)
also renders when `frozenVotes !== null`, drawing vote-lines from the dead game onto the new game's
player grid.

### Impact
Display shows incorrect vote arrows and vote-counts during `day` and `hunter_pending` phases in the
new game, until day_vote ends and `frozenVotes` is overwritten.

## Bug 3 — SECONDARY: Display `showResolution` / `resolutionState` Not Reset

### Location
`frontend-display/src/App.tsx:36-38` — `showResolution`, `resolutionState` state

### What Happens
If the host clicks "Play Again" while `showResolution === true` (mid-NightResolution interstitial),
`setGameId(newGameId)` is called. The display's `useGameState` clears `gameState`, but `showResolution`
and `resolutionState` are not reset. The display renders `<NightResolution>` with old game state
instead of the new game's lobby.

### Impact
Lower severity — the NightResolution interstitial auto-completes via `onComplete` after its animation
and the display recovers. But it creates a confusing flash of the old game during lobby setup.

## Proposed Fixes

### Fix 1 — Remove "Play Again" from mobile `GameOverScreen`
**File**: `frontend-mobile/src/components/GameOverScreen/GameOverScreen.tsx:68`

Remove the button entirely. Players are auto-redirected by the host's rematch. If they want to
leave, they close/refresh the app; bootstrap will either restore the game_over session or pick up
the redirect on WS reconnect.

Optionally: replace with a passive "Waiting for host…" label.

### Fix 2 — Reset `frozenVotes` on rematch in display `App.tsx`
**File**: `frontend-display/src/App.tsx`

Add a `useEffect` on `gameId` to reset `frozenVotes`, `showResolution`, and `resolutionState`:
```typescript
useEffect(() => {
  setFrozenVotes(null)
  setShowResolution(false)
  setResolutionState(null)
  prevPhaseRef.current = null
}, [gameId])
```

`prevPhaseRef` should also be reset here so the first phase transition in the new game is not
compared against the old game's final phase.
