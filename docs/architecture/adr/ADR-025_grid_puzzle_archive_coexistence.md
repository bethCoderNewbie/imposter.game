# ADR-025: Werewolf — Grid Puzzle and Archive Coexistence

## Status
Accepted

## Date
2026-04-15

## Context

PRD-013 introduces a 5×5 Data Node grid as a new night-phase mechanic for wakeOrder==0 players. A design decision was required: does the grid **replace** the existing Archive puzzle system, or does it **coexist** with it?

Three options were evaluated:

**Option A — Replace Archive:** The grid IS the new Archive. Villagers navigate nodes instead of receiving one static puzzle. The Archive tab disappears; the grid is the sole puzzle activity.

**Option B — Coexist (parallel systems):** The Archive puzzle runs as before on an ARCHIVE tab. The grid is an additional GRID tab. Players can use both during the same night phase.

**Option C — Replace + migrate hints:** Grid replaces Archive, and the three hint tiers fully replace all existing hint categories in `puzzle_bank.py`. Existing hint logic is removed.

The user selected **Option B — Coexist (parallel systems).**

---

## Decision

The Archive puzzle system (`player.puzzle_state`, `generate_night_puzzle()`, `submit_puzzle_answer` intent, `generate_hint()`) is unchanged. The Night Grid system (`player.grid_puzzle_state`, `generate_grid_layout()`, `select_grid_node` / `submit_grid_answer` intents, `generate_grid_hint()`) is additive.

The mobile night screen for wakeOrder==0 players shows two tabs:
- **ARCHIVE** — existing VillagerDecoyUI (one static puzzle, one hint)
- **GRID** — new GridMapUI (5×5 node navigation, multiple hints possible)

Both systems run independently during the same night phase. A player can solve their Archive puzzle for a hint AND navigate grid nodes for additional hints in the same round.

---

## Consequences

### Positive

**Zero regression risk.** The Archive system is not modified. All 236+ existing tests continue to pass without modification. Roles that currently use `puzzle_state` (Villager, Jester, Mayor via wakeOrder==0) are unaffected.

**Incremental rollout.** The grid can be toggled off (by not rendering the GRID tab) without touching the Archive system. A feature flag or game config option can gate grid access per game.

**Richer intelligence arc.** Players who engage with both systems can accumulate significantly more intelligence per night. The Archive provides guaranteed Tier 1 hints; the grid offers higher-tier hints at the cost of time and wolf radar exposure.

**Framer compatibility.** The existing `hack_archives` intent (Framer's Archive disruption) is unaffected. The Framer cannot hack grid hints — they operate on separate state fields. This means the Framer's power is unchanged while villagers gain new hint sources. A future PRD could add Framer grid interference if balance requires it.

### Negative

**Cognitive load.** Players have two simultaneous puzzle systems on their mobile night screen. First-time players may find the GRID tab overwhelming. UX mitigation: the ARCHIVE tab is the default on first night; the GRID tab is introduced with a brief tooltip.

**State model growth.** `PlayerState` grows by 3 fields (`grid_node_row`, `grid_node_col`, `grid_puzzle_state`). `NightActions` grows by 5 fields. The stripper must handle both puzzle stripping pipelines. Managed via `_strip_puzzle_for_player()` reuse and the `_build_na_for_view()` refactor (RFC-002).

**Two hint delivery pipelines.** `hint_reward` events are now generated by two code paths: `handle_submit_puzzle_answer` (Archive) and `handle_submit_grid_answer` (grid). The mobile client's hint display state (`latestHint`) must handle hints arriving from either source. Both use identical `hint_reward` WS event format — no client-side type changes needed.

**Night timer pressure.** With two parallel activities, the fixed night timer (60s default) may feel tight for players attempting both Archive and multiple grid nodes. This is an intentional design trade-off: the grid creates opportunity cost, not guaranteed benefit. If playtesting reveals consistent frustration, `GameConfig.night_timer_seconds` can be tuned per difficulty level.

### Option A Rejection Rationale

Replacing the Archive would require:
- Migrating all existing `puzzle_state` handler code to the new grid paths
- Rewriting ~40% of `stripper.py` view logic
- Invalidating ~30 existing backend tests that assert on Archive puzzle behavior
- Breaking the `submit_puzzle_answer` intent contract (used by existing clients)

The risk of regression and the breadth of changes is disproportionate to the benefit. The Archive system is proven, tested, and balanced. The grid adds value on top of it; it does not require replacing it.

### Option C Rejection Rationale

Migrating all hint categories from `generate_hint()` into grid-only generation would mean villagers receive zero hints in nights when they fail to complete any grid nodes. The Archive provides a baseline guarantee: solve the Archive puzzle, receive at least one hint. Removing this guarantee increases variance in a way that may frustrate less engaged players. Coexistence preserves the Archive's safety net while the grid offers upside.
